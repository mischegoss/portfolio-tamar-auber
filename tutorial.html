The Overview

ARIA Integration — System Overview

Helping new developers understand the "why" and "how" of the system before they start building.

Reading time: 4 minutes
Doc type: Conceptual overview

Learning objective: Understand how ARIA connects a chat interface to a workflow automation platform — so users can trigger backend processes through conversation and get results without leaving their chat window.

Prerequisites

While not required to build or understand the integration, familiarity with the following concepts is helpful:

  • APIs — Systems exchange data through structured requests and responses
  • Webhooks — Event-driven HTTP requests that push data between systems
  • JSON — Structured data format for API communication
  • Parameters — Discrete pieces of information collected from users (username, ticket number, etc.)

Technical implementation details — including specific field names, values, and step-by-step configuration instructions — are covered in subsequent sections.

What This Integration Accomplishes

ARIA connects HelixBot (a conversational AI platform) to FlowEngine (a workflow automation platform). A user types a request in chat, HelixBot collects what it needs, and FlowEngine runs the workflow — all without the user touching another system.

ℹKey capability: Users interact with enterprise systems and workflows using natural language in HelixBot, without leaving the chat window.

How HelixBot and FlowEngine Work Together

HelixBot is the front end. It interprets what the user is asking, asks any follow-up questions, and packages the user's input into a structured request.

FlowEngine is the back end. It receives that request, runs the workflow, and sends the result back to HelixBot to display in chat.

The two systems talk to each other via webhook. HelixBot sends a JSON payload to FlowEngine's API, holds the conversation open, and displays whatever FlowEngine returns. The user sees only the result — the plumbing is invisible.

The Complete Flow

Here's what happens when a user types "I need to reset my password":

#StageWhat happens
1 User Trigger The user types a request. HelixBot's intent prediction matches it to the "password reset" workflow.
2 Information Gathering HelixBot collects required data through conversation or user profile data. The user answers any prompted questions. These answers become the input parameters sent to FlowEngine.
3 Data Transformation HelixBot packages all inputs and session context into the JSON format FlowEngine expects. A mapping script handles the translation automatically.
4 Webhook Sent HelixBot sends the structured payload to FlowEngine's Events API endpoint and holds the conversation open, waiting for a response.
5 Workflow Execution FlowEngine receives the request, routes it to the correct workflow, executes it, and packages the result.
6 Callback & Display FlowEngine sends the result back to HelixBot using the callback URL. HelixBot identifies the correct conversation and the response appears in the user's chat: ✓ Password reset successfully.

⏱Total time: Typically 5–15 seconds from start to finish.

What You Can Build

This integration supports any workflow that collects structured input through conversation, executes a FlowEngine workflow, and returns a confirmation or result to the user.

Common use cases

  • Password resets and account unlocks
  • Software access requests
  • Approval workflows
  • System status checks

In this version: Clone and customize the pre-built templates for each new use case. Each intent in HelixBot corresponds to a workflow in FlowEngine. Most implementations take 30–60 minutes per workflow, and the consistent template structure ensures every integration behaves the same way.

The Playbook

Configure the ITSM Platform Connection Back to HelixBot

A step-by-step setup guide with a built-in troubleshooting section.

Doc type: Procedural guide
Estimated time: 15–20 minutes

At this point, HelixBot can send requests to the ITSM Platform. This guide sets up the other direction — so the ITSM Platform can send data back to HelixBot. You'll work entirely inside the ITSM Platform interface. Before you start, confirm that the prior setup step (committing the Helix update set) is complete.

Prerequisites

  • HelixBot configuration completed
  • Helix update set committed in the ITSM Platform
  • ITSM integration user password on hand
  • HelixBot tenant URL on hand

Step 1: Log In to the ITSM Platform

⚠Important: All steps below happen in the ITSM Platform — not in HelixBot.

  1. Open your ITSM Platform instance in a browser.
  2. URL format: https://[instance].itsm-platform.com
  3. Log in with your ITSM Platform admin credentials.

Step 2: Open the Helix Integration Settings Module

  1. In the ITSM Platform Navigator (left sidebar search bar), type: Helix
  2. In the results, locate the Helix application modules.
  3. Click Helix > Integration Settings.
You should see a form titled "Integration Settings."

Step 3: Enter HelixBot Connection Credentials

In the Integration Settings form, enter the following:

FieldValue
Usernamehelix.integration
PasswordThe integration user password from your prior setup step
Base URLhttps://{tenant}.helix-platform.com/

⚠Don't forget the trailing slash. A missing trailing slash is the single most common cause of connection failure.

✅ Correct: https://acme.helix-platform.com/
❌ Incorrect: https://acme.helix-platform.com

Click Save or Submit.

Settings should save without errors.

Step 4: Open the Helix Integration Hub Module

  1. In the ITSM Platform Navigator, type: Helix
  2. Click Helix > Integration Hub.
You should see a screen with a Test Connection or Test/Refresh Connection button.

Step 5: Test the Connection

  1. Click the Test/Refresh Connection button.
  2. Wait 5–10 seconds while the ITSM Platform sends a test request to HelixBot.
  3. Look for a success message in the interface.

✅Bidirectional connection complete. You have successfully configured both directions:

✅ HelixBot → ITSM Platform
✅ ITSM Platform → HelixBot

The two systems can now communicate bidirectionally for ticket creation and data exchange.

Troubleshooting

Connection test fails — Missing roles Cause: integration user missing required roles
  • Verify all three Helix roles were assigned to the integration user.
  • Confirm the user also has the itil role if required by your instance configuration.
  • Verify Password needs reset is unchecked on the user record.
Connection test fails — Authentication error Cause: incorrect username or password
  • Verify the username is exactly: helix.integration
  • Verify the password matches what was set during integration user setup.
  • Verify Password needs reset is unchecked.
Connection test fails — URL error Cause: missing trailing slash

A missing trailing slash is the most common cause of connection failure.

  • ✅ Correct: https://acme.helix-platform.com/
  • ❌ Incorrect: https://acme.helix-platform.com
  • The URL must include https:// and must end with a trailing slash.
Connection test fails — Network error Cause: firewall or network configuration
  • Verify the ITSM Platform instance can reach external URLs.
  • Contact your network admin if needed.
Cannot save settings Cause: wrong application scope

You may be working in the Global scope instead of the Helix scope.

  • Look for a banner at the top of the screen prompting you to switch application scope.
  • Click the link to switch to the Helix application scope.
  • Re-enter settings and save.
No response from test Cause: HelixBot tenant unreachable
  • Verify the HelixBot tenant is accessible by opening it directly in a browser.
  • Wait 1–2 minutes and try the test again.
Helix modules not visible Cause: update set not committed or browser cache
  • Return to the prior setup step and verify the update set status shows Committed.
  • Hard refresh your browser (Ctrl+Shift+R).
  • Log out and back in to the ITSM Platform.
